Wiom CSP App · Designer's Reference

The CSP's home page: what they see, what they do, and when.

A design decision guide — no code. How the partner reads their work queue, how cards communicate urgency, what each task journey feels like, and where the current design helps or fails them.

The home page is the CSP's entire workday in one scroll.

A CSP partner (field technician / shop owner) opens this app 15–30 times a day. The home page answers one question: "What do I do next?" Everything below is designed to help them triage — scan, pick the most urgent task, act, come back.

R1

What the CSP sees, top to bottom

1. Purple header bar

Business name + role (मालिक / member). Hamburger for drawer (wallet, team, support). Always visible, never scrolls.

Message to CSP: "This is your workspace. You're logged in."

2. Scorecard strip (4 chips)

Connections (142/160), earnings (₹18,400), quality (● अच्छी), work status (खुला). Pink dot if something changed since last visit.

Message to CSP: "Here's how you're performing overall."

3. Urgent alerts (banners)

Yellow/orange warnings like "2 नेटबॉक्स की पुष्टि बाकी · 31घं बचे". Info notices from the company. Only 1 at a time; extras say "+2 more".

Message to CSP: "Stop. This needs attention before anything else."

4. Task feed (the core)

Individual cards for each job — installs, restores, pickups, recharges. Sorted by the server (most urgent first). Each card = one job, two lines, no buttons.

Message to CSP: "Here's your work queue. Tap to act."

The fundamental pattern → The home page is a read-only triage surface. Cards show information and urgency. All action happens after a tap — inside drilldowns, sheets, or wizards. There is no inline editing, no swipe-to-act, no buttons on cards.

R4

What the CSP can NOT see or do from the home feed

Cannot act on the feed

No buttons, no swipe actions, no inline edits. Every action requires tapping into a card first. The feed is read-only.

Cannot see live updates

If the server changes a task while the CSP stares at the screen, nothing updates until they pull-to-refresh or tap something. There is no real-time push.

Cannot reorder tasks

Server decides the order. No filters, no search, no sort options. The CSP sees what the server prioritises.

Cannot act offline

All action buttons are disabled offline. A warning banner appears: "इंटरनेट नहीं है". CSP can still read the cached feed.

What can go sideways

Escalation

If CSP takes too long, complaint escalates. "एस्केलेट हुआ — आपकी कार्रवाई ज़रूरी" (CSP can still act) or "Wiom ने ले लिया है" (platform takeover, CSP locked out).

Customer denies fix

"ग्राहक ने मना किया" — customer says it's not fixed. CSP can retry proof or launch a swap.

Delegation + overdue

If teammate is assigned but late: "समय निकल गया" (red). CSP can recall the task and take over, or reassign.

SLA timer → A deadline pill shows urgency on the card. Under 4 hours = amber. Past deadline = red "देरी हो गई". Unlike install timers, this one does NOT disappear at zero — it stays red as a permanent warning.

How the app talks to the CSP.

Every piece of the home page is a communication channel. Here's what each signal means and how they combine to create a sense of urgency, progress, or safety.

S1

The 4 emotional tones

The app uses 4 tones across banners and drilldowns. Each has a distinct background, icon colour, and emotional meaning:

⚠
Warning — "You need to act"
Something needs CSP attention. A deadline is approaching, a slot hasn't been picked, a complaint needs acceptance. The CSP should act now or soon.
ℹ
Info — "Something is in motion"
The system is processing, someone is working on it, verification is running. CSP doesn't need to act. Just awareness that things are moving.
✓
Positive — "This went well"
Slot confirmed, technician assigned, installation complete, money earned. Confirmation and reward. The good news.
✕
Negative — "Something went wrong"
Deadline passed, teammate overdue, customer denied fix, installation failed, escalation. The red flag. Often means CSP must intervene to fix it.
Important → The drilldown banner (inside the task detail) uses these 4 tones with hardcoded rules per state. The feed card (on the home scroll) uses a different colour system — the stripe + line-2 tint comes from the server, not from these tones. They are independent signals. This means a card on the feed can look "normal" but the drilldown can show a warning.

S3

When the app goes silent (and the CSP is left guessing)

There are moments where the app stops communicating and the CSP has no feedback. These are design pain points:

No real-time updates

If a task changes server-side while CSP is looking at the feed, nothing happens on screen. No push, no poll, no indicator. CSP must pull-to-refresh or re-open the app to see changes.

CSP experience: "Did anything change? I don't know until I swipe down."

Passive states have no progress

When waiting for customer confirmation or system verification, the card just sits there with static text. No progress bar, no "checking..." animation, no ETA.

CSP experience: "Is the system actually doing something? How long will this take?"

Delegation has limited visibility

When CSP delegates to a teammate, they only see "not started" → "in progress" → "overdue". No intermediate steps, no progress percentage, no ETA from the teammate.

CSP experience: "My teammate is 'working on it' — but is he at step 2 or step 9?"

S4

How the "instant response" illusion works

When the CSP taps an action button, the card changes instantly — before the server even knows. This is the "optimistic update" pattern. Here's what the CSP experiences:

What CSP sees

1. Taps "Acknowledge" on a pickup

2. Card instantly changes to "work started" — stripe colour may change, line-2 text updates, CTA disappears

3. Toast appears: "acknowledged"

4. Nothing else visible happens (server confirms in background)

It feels instant. No spinner, no "saving..." state.

What happens if it fails

If the network call fails:

1. The card silently reverts to its previous state on the next feed refresh

2. An error toast appears

Design gap: There's no explicit "failed to save" state. The card just goes back to where it was, which can be confusing if the CSP already moved on mentally.

What this means for design decisions.

Synthesis of everything above into actionable design context — the summary split of action vs passive, where the current UX helps or fails, and what constraints you're working within.

D1

The action vs passive summary

Across all task types, here's what the CSP experiences. Use this as a lens for any redesign: action states are where the UX matters most — that's where the CSP makes decisions.

🔴 CSP must act (15 moments)

Install: Pick a slot · Assign executor · Start the wizard · Re-pick slot if failed · Change technician if overdue

Restore: Accept complaint · Start work · Run proof steps · Resolve · Handle escalation · Handle customer denial · Recall overdue teammate

Pickup: Acknowledge · Take photos / scan serial · Resubmit if rejected

Shifting: Accept + complete the shifting

Recharge: Select customers · Confirm + claim commission

🟣 CSP waits (10 moments)

Install: Customer choosing slot · Scheduling system working · Wizard submitted · System verifying

Restore: System verifying · Platform takeover (locked out)

Pickup: Photos under review · Verification pending

Shifting: Customer confirming · SLA escalated

Recharge: Auto-recharge processing

🟡 CSP monitors teammate (5 moments)

Install: Teammate hasn't started · Teammate working · Teammate overdue

Restore: Technician assigned · Teammate not started / in progress

Can see status + call teammate. Can recall/reassign only when overdue.

D2

Where the current design helps the CSP

✅ Urgency is instantly scannable

The stripe + tint system means a CSP can scan 10 cards in 2 seconds and spot the urgent ones. Red/orange stripes jump out. This is the strongest design choice on the page.

✅ Actions feel instant

Optimistic morphs mean tapping "Acknowledge" feels like 0ms. No spinners, no waiting. The CSP moves fast through their queue.

✅ Server-driven flexibility

Text, colours, order, and CTA config all come from the server. The operations team can change messaging, urgency, and priority without waiting for an app release.

✅ One card type renders everything

The generic card renderer means new task types can be added without any app change. The CSP sees a consistent visual language across all work types.

D3

Where the current design fails the CSP

❌ No way to distinguish action from passive on the feed

A card that needs CSP action looks identical to a card where they're just waiting. Both have the same two-line layout, same card shape, same tap behaviour. The CSP must tap into every card to find out if they need to do something.

Impact: CSPs waste taps. They open drilldowns for tasks where there's nothing to do, then back out. With 15+ cards, this adds up.

❌ No progress indicators for waits

When waiting for verification, customer confirmation, or system processing, the card shows static text. No spinner, no progress bar, no "estimated time". The CSP has no idea if it's been 5 seconds or 5 hours.

Impact: CSPs pull-to-refresh anxiously during wait states, wondering if the system is stuck.

⚠️ No filtering or search

CSP can't filter by task type (show only pickups), status (show only urgent), or search by customer/ID. With 15+ cards, finding a specific task means scrolling through everything.

Impact: When a customer calls asking about their specific install, CSP scrolls through the entire feed to find it.

⚠️ Stale data with no awareness

The feed only refreshes on pull or action. No background refresh, no "last updated X min ago" indicator. CSP doesn't know if they're looking at fresh or 30-minute-old data.

Impact: CSP might miss time-sensitive changes. A pickup deadline could pass without the card updating.

⚠️ Delegation is a black box

When work is delegated, CSP sees 3 states: not started → working → overdue. No intermediate steps from the 33-screen wizard, no progress percentage, no location of the teammate.

Impact: CSP can't help a stuck teammate or gauge if they'll finish on time.

⚠️ Silent failure on optimistic rollback

If an action fails after the card already visually updated, it quietly reverts. The CSP might think they acknowledged a pickup but it didn't actually go through.

Impact: CSPs may miss failed actions, leading to missed SLAs.

The core design question → The home page is built for triage — scan, pick the most urgent task, act, come back. The server drives priority and urgency. The app renders a generic card for everything. This architecture is powerful for flexibility but weak for differentiation — all cards look the same, action vs passive isn't visible, and wait states feel dead. A redesign can address these gaps without breaking the server-driven architecture — add visual cues for "your action needed" vs "waiting", add progress indicators for passive states, and add a "last refreshed" timestamp. These are all client-side rendering changes that don't touch the API contract.
