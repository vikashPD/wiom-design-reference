Split the feed: "Your actions" on top, "In progress" below.
Instead of one mixed list, separate cards into things CSP must do and things that are being handled. The CSP opens the app and instantly sees only their pending work at the top.

What changes

"आपका काम" section at the top

Cards where CSP must act — propose slot, accept complaint, acknowledge pickup, start work. Sorted by urgency within this group.

"हो रहा है" section below

Cards where system, customer, or teammate is handling it — verification pending, customer confirming, delegated work. Slightly dimmed to show lower priority.

Counts in section headers

"3 बाकी" tells CSP exactly how many actions they have before scrolling. When count = 0, the section shows "कोई काम नहीं" (no actions) — the reward for finishing everything.

How it classifies cards

The classification rule

The server already sends isPassive and isCspActionable on every task. The rule is simple:

Action section: isPassive = false AND the card has a CTA action

In-progress section: isPassive = true OR no CTA available OR state is a wait/monitor state

If a delegated task becomes overdue, it moves from "in progress" to "action" — because now the CSP needs to intervene.

Feasibility

Effort
Low
Client-side sort in HomeScreen.kt
Impact
High
CSP sees action count in 0.5 seconds

What's needed

App change only. Sort tasks into two lists before rendering. Add section headers. No API change — all classification data already exists in the feed.

Risk

Overrides server sort order within each section. Passive cards get less attention — fine for verifications, but escalations that flip from passive to action need to move up immediately on next refresh.

What makes the hero card different from regular cards

Distinct visual treatment

Pink gradient background, larger padding, branded border. It looks different from every other card — the CSP's eye goes here first, every time.

"अभी करें" label

Hindi label at the top: "Do now". Removes ambiguity — this isn't information, it's a call to action.

CTA button on the card

The primary action button is right here — no need to drill down. "स्लॉट चुनें →" opens the slot sheet directly. One tap to act.

Context line

"ग्राहक इंतज़ार कर रहा है" — explains why this is urgent, not just what the task is. Motivates action.

"+N और काम बाकी" counter

Below the hero: tells CSP how many more action items are waiting. Creates a sense of queue progress.

Completion state

When all actions are done, the hero space shows a positive message: "✅ सब काम हो गया" (all done). The reward for clearing the queue.

How it picks the hero

The hero is the first card from the action group (sorted by server urgency). Priority order:

1. Overdue tasks (red stripe, OVERDUE threshold) — deadline already passed

2. Urgent tasks with live countdown (P41/P74/pickup SLA timer running)

3. Tasks needing immediate acceptance (PENDING_ACCEPTANCE, RESPOND_NOW)

4. Regular action tasks (slot proposal, executor assignment, etc.)

When CSP completes the hero action, it morphs optimistically → the next highest-priority task slides into the hero spot.

Feasibility

Effort
Medium
New composable + classification logic
Impact
Very High
#1 priority visible in 0 seconds

What's needed

New HeroCard composable. Client-side priority classification. CTA routing from the hero card (reuses existing sheet/drilldown logic). Context line copy — ideally server-sent, or client-generated from existing fields.

Risk

Only shows 1 task. CSPs with 2 equally urgent tasks might miss the second. Mitigated by the "+N more" counter and the regular feed below. Also: the hero takes vertical space — less room for the rest of the feed on small screens.

The three marker types

स्लॉट चुनें → Action chip

Pink chip with arrow. Shows the specific action needed: "स्लॉट चुनें", "शुरू करें", "स्वीकार करें". Uses existing ctaLabelKey from server. CSP knows exactly what to do without opening the card.

⟳ वेरिफाई Wait chip

Grey chip with spinner-style icon. Communicates: "system is handling this, you don't need to do anything." Uses short version of status text.

👤 राम Delegated chip

Purple chip with person icon + teammate name. Communicates: "your team member is on it." Uses existing executorName from server.

Feasibility

Effort
Very Low
Add one chip to TaskCard.kt
Impact
Medium
Scannable but doesn't change structure

What's needed

A single conditional chip in the card renderer. Server already sends ctaLabelKey, isPassive, executorName. No API change. Smallest possible design change.

Risk

Line-1 is already dense (icon + type + ID + locality + badge). Adding a chip could cause text truncation on small screens. Needs careful sizing and overflow handling.

The four sections

🔴 अभी करें — Do now

Overdue tasks + tasks with active countdown timers + tasks needing immediate response (PENDING_ACCEPTANCE, RESPOND_NOW). These have crossed or are approaching a deadline.

Feeling: Urgency. "These can't wait."

🩷 आज करना है — Do today

Action-required tasks without immediate urgency — slot proposals with plenty of time, executor assignments, work that can happen within the day.

Feeling: Important but not panicking. "Get to these when you can."

🟡 टीम पर — On team

Delegated to a teammate. CSP can see status but can't directly act. Shows teammate name. If a teammate goes overdue, the card moves UP to "अभी करें".

Feeling: Awareness. "Someone's handling it, I'm just keeping an eye."

⚫ इंतज़ार में — Waiting

System verification, customer confirmation, processing. Slightly dimmed. CSP can safely ignore these until they move.

Feeling: Peace. "Nothing for me to do here."

The "at a glance" moment → When CSP opens the app, they read the section headers alone: "2 ज़रूरी · 1 आज · 1 टीम · 1 इंतज़ार". That's 5 numbers that tell the entire story of their workday. Without reading a single card.

Feasibility

Effort
Medium
Classification logic + dynamic section rendering
Impact
Very High
Full workday picture in 1 second

What's needed

Classification logic to bucket tasks into 4 groups using existing fields (thresholdState, isPassive, executorName, isCspActionable). Section header composable with counts. Opacity dimming for lower priority sections.

Risk

More sections = more scrolling. If CSP has 10 urgent tasks, the "do today" section pushes way down. Empty sections should collapse entirely. Also: 4 categories is more cognitive load than 2 (Idea 1). For some CSPs, "urgent" vs "today" distinction might be confusing.

IDEA 05
Inline CTA: put the action button directly on the card.
Break the "no buttons on feed" rule. For simple, unambiguous actions, show the CTA right on the card. Cards with buttons = "do something". Cards without buttons = "just watching." The visual split is instant and obvious.

Which actions are "simple enough" for inline

The rule: if the action opens a sheet or direct action, it can be inline. If it requires a full drilldown, it can't.

Can be inline: "स्लॉट चुनें" (opens sheet) · "स्वीकार करें" (direct action) · "शुरू करें" (direct action) · "Acknowledge" (direct action)

Should NOT be inline: "Start wizard" (opens 33-screen flow) · Complex proof steps · Anything requiring context from the drilldown

The server already flags this via executionMode: DIRECT and ACTION_SHEET can be inline. DRILLDOWN_REQUIRED stays drilldown-only.

Feasibility

Effort
Medium-High
Card layout change + tap target management
Impact
Very High
Action vs passive is visually obvious + saves 1 tap

What's needed

Third line in TaskCard composable. CTA routing from the card level (reuses existing morph + sheet logic). Separate tap targets for card body (→ drilldown) vs CTA button (→ action). Passive status line for non-action cards.

Risk

Cards become taller (3 lines instead of 2). Less cards visible per screen. Accidental CTA taps are possible — needs adequate spacing. The "no buttons on feed" design principle was intentional — breaking it changes the character of the page from "triage list" to "action list".

The biggest payoff → This is the only idea that both visually differentiates action from passive AND reduces taps. The CSP can acknowledge a pickup, accept a restore, and propose a slot — all from the feed, without opening a single drilldown. Three actions that currently take 6+ taps → 3 taps.

How they combine: the recommended stack

These ideas aren't mutually exclusive. Here's how they layer from least to most ambitious:

Quick win — ship this week

Idea 1 (split feed) + Idea 3 (action markers)

Split into 2 sections with counts + add action/passive chips on each card. Together, these give the CSP two levels of clarity: section counts for the big picture, chips for per-card scanning.

Low effort. High impact. No API change.

Full vision — next milestone

Idea 4 (smart sections) + Idea 2 (hero card) + Idea 5 (inline CTA)

4 urgency groups give the full workday picture. Hero card at top for #1 priority with a direct CTA. Inline buttons on action cards save taps. Passive cards show spinner/person status.

Medium effort. Very high impact. Transforms the home page from triage list to action dashboard.
