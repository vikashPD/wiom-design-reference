Design takeaway → Almost everything below the header is a generic renderer + server schema — chips, task cards, batch blocks, banners and their colours, labels and routes all arrive from the backend. Most “home page” changes are backend/config changes, not app releases. Redesigning a surface here is designing a contract as much as a layout.

How every task card behaves.

The four kinds of work — Installation, Service, Recharge, NetBox Pickup — documented twice: what the user does, and how the app implements it. Start with the shared engine: one generic card renders all four; the server sends the state, the app renders it.

User behaviour App / implementation good attention overdue / failed

The shared task-card engine

Every card is drawn by one composable from one model (TaskDetail). No task type has a bespoke card. The card is a signal, not a surface for action: two lines, no button. All acting happens after a tap.

The contract

Server-driven (schema)

Task family, state, icon key, label key, the reason-line template (with {name}/{time}/{remaining}), threshold state, accent colour, unread count, deadlines, execution mode, auto-dismiss. Add a new family with zero app release — taskFamily is a raw string, not an enum.

Client-side (rendering only)

Icon-key→vector map, accent→colour, threshold→colour, short-id, countdown copy. Everything hardcoded is a rendering of a server key — no labels, colours, or states invented locally.

Grouping & tap routing

Grouping (HomeScreen :376–464)

Dedupe by id → pickups always grouped, even one. RECHARGE & NETBOX_RETURN render as collapsed batch cards. Others: inline rows ≤6, “view all” >6. Always server order.

Tap routing (HomeViewModel)

Body tap → fetch detail → drilldown (NETBOX_RETURN → pickup wizard). CTA tap switches on executionMode: DIRECT in place, ACTION_SHEET a sheet by state, DRILLDOWN_REQUIRED full screen.

Unread badge

Pink count = latest − acked version. Cleared on drilldown dismiss; a later server bump re-raises it.

Auto-dismiss

Server autoDismiss + dismissAfterSeconds → card removes itself; keyed on id so recomposition doesn't restart it.

Installation

INSTALL

Book a slot, assign who does it, then run a guided on-site wizard — arrival, KYC, payment, device photos, provisioning, Happy-Code OTP. One task object drives a card, a drilldown, and a ~33-screen wizard.

user

What the partner does

Stage A · inside the drilldown

  1. 1
    Propose a slot
    Today or Tomorrow only; submitting is implicit acceptance, time arranged offline.
  2. 2
    Assign an executor
    Self + team, self pre-selected. Self → you run the wizard; teammate → DELEGATED_*, only they run it.
  3. 3
    Start / resume the wizard
    Launches or resumes at the last completed step.

Stage B · the on-site wizard

  1. 4
    PAYG consent → arrival
    PAYG audio + confirm, then “I've arrived” marks the visit (→ ARRIVED_AT_SITE).
  2. 5
    Selfie → Aadhaar KYC
    Technician selfie, then Aadhaar front + back. App re-fetches and branches on payment.
  3. 6
    Security fee → power-up
    Payment checklist polls until PAID, then 30-second power-up + LED confirm.
  4. 7
    Customer details → ISP account
    Create the ISP account via self-portal / WhatsApp / call.
  5. 8
    ISP form → placement → device photos
    Device ID + PPPoE (BE-verified), signal check, NetBox / 3-pin / wiring photos.
  6. 9
    Provisioning → optical / speed

  7. 10

ISP Recharge

RECHARGE

Renew customers' plans and claim the partner's commission. Surfaces as a collapsed batch card. Two flows, chosen by renewalFlowType: a standard confirm-and-claim flow, and an auto-recharge flow where the app drives the ISP portal itself.

user

Standard flow

  1. 1
    Education gate
    First-time explainer; skipped once acknowledged.
  2. 2
    Customer list
    Pending renewals split into a 50→100 Mbps migration group and a regular group. Multi-select.
  3. 3
    Confirm → calculate
    Confirm “recharge N customers?” → app calculates commission earned.
  4. 4
    Claim commission
    Release commission; list refreshes — if none pending, returns home.
No in-app customer payment. The standard flow doesn't collect money — the partner asserts the recharge happened and the app calculates → releases the partner's commission (amounts in paise). The customer's payment happens out of band.
user

Auto-recharge flow

  1. 1
    Connection list
    Pending claims; check for a saved ISP portal.
  2. 2
    Validate ISP login
    Partner validates portal URL + credentials (handles low balance).
  3. 3
    Trigger renewal + poll
    App performs the renewal on the ISP portal, then polls up to 24× every 5s.
  4. 4
    Collect commission
    Shows earned + session total, returns to the list.

Two colour signals, not one

The feed card and the drilldown banner are coloured by different rules. The card's colours are server-driven (accentType + thresholdState). The banner's are hardcoded per state — that's the deterministic state→colour map. Kept independent on purpose (INV-H-08).

The feed card, per state

The banners had a colour for every state. The card does not. The stripe and reason line are painted from server fields, so there's no hardcoded state→colour map on the card. What the code decides per state is the icon and which line-2 mode shows — plus grouping. So this shows the distinct card renderings precisely, then a per-state table of the line-2 mode.

Server-driven vs code-driven. On the card, the stripe (accentType) and the reason text + tint (reasonDisplayTemplate + thresholdState) come from the backend. Specimens render the code-deterministic parts precisely (icon, line-2 mode, timer) and the server parts as their typical appearance.
