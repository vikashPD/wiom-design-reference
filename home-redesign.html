Wiom CSP App · Home Redesign

Action-First Home Feed

Split the home into two clear zones — "आज के काम" (today's tasks) on top, "चल रहा है" (in motion) below. CSP opens the app and instantly knows what to do.

1. Action first

Everything that needs CSP action right now goes in the top section. If the CSP doesn't need to do anything — it's not in "आज के काम".

2. Cards move dynamically

When a non-action card becomes actionable (customer confirms slot, photo rejected), it moves up to "आज के काम". When CSP completes an action, it drops to "चल रहा है".

3. Recharge on top, always

ISP Recharge batch stays pinned at the top of action section — quickest win, highest retention impact, easiest to complete.

How cards move between sections — real examples

Cards are living — they jump between "आज के काम" and "चल रहा है" as state changes

🏠 Install — slot confirmed

CSP proposed a slot → card drops to "चल रहा है" (waiting on customer). Customer confirms the slot → card jumps back to "आज के काम" with "Assign technician" CTA.

Action → Motion → Action

🔧 Service — delegated overdue

CSP assigns technician → card drops to "चल रहा है" (tech assigned). Technician misses deadline → card jumps back to "आज के काम" with red "Delegate overdue" urgency.

Action → Motion → Action (urgent)

📦 Pickup — assign then delegate

New pickup arrives as individual card in "आज के काम". CSP assigns technician → card moves to batch in "चल रहा है". If tech goes overdue → individual card pops back into "आज के काम".

Individual → Batch row → Individual (overdue)

⚡ Recharge — always on top

Recharge batch card never moves — always pinned at position #1 in "आज के काम". Count updates as CSP completes recharges. Card disappears only when all pending = 0.

Pinned #1 → count decrements → Gone when 0

🚚 Shifting — verify comes back

CSP assigns tech → card drops to "चल रहा है". Work completed, needs verify → card jumps back to "आज के काम" with "Verify" CTA. After verify → drops to "Verification pending".

Action → Motion → Verify → Pending

🏠 Install — resolved

Installation verified → card shows briefly as "Resolved ✓" in "चल रहा है" → auto-removes from feed after ~5 seconds. Count in section header decrements.

Verifying → ✓ Gone

⚡ Recharge batch — always on top

Why recharge gets the #1 position and how the batch card works

Why #1?

  • Quickest action — collect money, tap recharge, done
  • Highest volume — CSP may have 10-20 pending daily
  • Direct retention impact — expired plan = customer leaves
  • Revenue impact — CSP earns commission per recharge
  • Low effort, high reward — phone call + collect + tap

Batch card behaviour

  • Always visible in "आज के काम" if pending > 0
  • Shows total count prominently: "12 Recharge pending"
  • Sorted by deadline — most urgent customer on top
  • Orange accent to stand out from pink action cards
  • Single CTA: "Recharge" opens the recharge flow
  • Count decrements in real-time as CSP completes
  • Disappears when all recharges done (pending = 0)

Priority sort order — inside "आज के काम"

How cards are ordered after the pinned recharge batch

Priority What Logic
#1 ⚡ Recharge batch — pinned Always first. Never moves. Shows total pending count.
#2 OVERDUE cards — any task type Deadline passed. Red cards. Pickup delay, Delegate overdue, Work delay. Most overdue first.
#3 URGENT cards — any task type Deadline approaching. Orange cards. "Due in X hours". Closest deadline first.
#4 NORMAL action cards — any task type No urgency pressure. Sorted by most recently updated first — latest server update on top.
Sort within same urgency level: Most recently updated card on top. This means if a customer just confirmed a slot, that install card jumps to the top of its urgency group — CSP sees the freshest actionable item first.

Implementation feasibility

What changes in the current architecture

Server-driven — no app release needed

  • Section assignment (action vs motion) can be driven by existing isCspActionable flag from backend
  • Priority sort uses existing ThresholdState + updatedAt fields
  • Recharge pinning is a client-side sort rule — "if RECHARGE, always position 0"
  • Section header counts are client-side computed from feed data

Needs app change

  • Pickup split logic — client needs to render pickup as individual card (action) or batch row (motion) based on state
  • Two-section LazyColumn — replace flat feed with sectioned layout
  • Section headers — new sticky header composable with count badge
  • Card movement animation — optional but adds polish when cards shift between sections
