Step 3 · The solution

Single feed — latest update on top, red dot for action

One unified feed sorted by most recent update. Cards needing CSP action show a red dot — the dot stays until the CSP acts. All cards (pending, in-process, completed) live in the same feed and naturally drift down when they have no new updates.

Interactive prototype below — tap any card to open the detail screen, take action, and watch the red dot disappear.

1. Latest update = top

Whenever a card gets a new update (state change, customer action, system event), it jumps to the top of the feed. The freshest information is always first.

2. Red dot = action needed

Cards needing CSP action show a red dot. The dot stays until the CSP acts. No dot = FYI, CSP can see it but doesn't need to do anything.

3. Natural decay

Cards with no new updates drift down as other cards with fresh updates push above them. After 7 days, cards auto-remove (backend-controlled).

Home Feed — 5 Card States

Every card on the home feed is always in one of these 5 states

Core rule: The home feed is a single sorted list. Every card lives here in one of 5 states. Cards with action + urgency sort to the top. Cards with no updates naturally drift down. The server drives all state changes — the app only renders.
Batch cards (NetBox Pickup & ISP Recharge) behave like all other cards. They are no longer pinned at the top or grouped into batch rows. Each Pickup and each Recharge is an individual card in the feed — it receives updates, moves to the top when updated, shows a red dot when action is needed, escalates through Overdue Soon → Action Delayed like any other card, and drifts down when idle. The same 5 states below apply to all card types equally — Install, Service, Pickup, Shifting, and Recharge.

Sort Priority — top to bottom

How cards are ordered in the feed at any given moment

# Card State Server signal Visual indicator
1st Action Delayed (overdue) OVERDUE + isCspActionable Red dot + red badge "देरी हो गई"
2nd Action Overdue Soon (urgent) URGENT + deadlineAt Red dot + amber countdown ticking
3rd Update with Action (normal) NORMAL + isCspActionable Red dot, no urgency badge
4th Normal Update (FYI) NORMAL, no action No red dot, sorted by recency
5th Card Removed (terminal) Terminal state Greyed out, drifting to bottom
Within each priority level: most recently updated card comes first. This means if two cards are both "Action Delayed", the one that was updated most recently sits higher.

Overdue Soon and Action Delayed are not separate states in this table — they are urgency escalations of the Action state. The same "Assign self/tech" state can be Normal Action, Overdue Soon (< 4h left), or Action Delayed (deadline passed) depending on the server's thresholdState value.

Server signals — what drives each state

All state determination is backend-driven. The app only renders.

Server field Values What it controls
isCspActionable true / false Whether to show the red dot. true = action needed. false = FYI.
thresholdState NORMAL / URGENT / OVERDUE Urgency level. NORMAL = no urgency. URGENT = < 4h left, amber. OVERDUE = deadline passed, red.
accentType normal / warning / negative Left stripe color on the card.
deadlineAt ISO timestamp SLA deadline. Client ticks a 1-second countdown loop against this for the live timer display.
updatedAt ISO timestamp Last update time. Determines card position in the feed — most recent on top.
Key insight from codebase: The two signals (stripe via accentType and urgency via thresholdState) are set independently by the server. A card can have a grey stripe with an urgent line. The app never calculates urgency — it receives the state and renders it.

Implementation feasibility

What changes in the current architecture

Already available from server

  • isCspActionable — drives red dot (exists today)
  • thresholdState — NORMAL / URGENT / OVERDUE (exists today)
  • accentType — left stripe color (exists today)
  • deadlineAt — SLA deadline timestamp (exists today)
  • updatedAt — last update timestamp (exists today)
  • 1-second client-side countdown loop (exists today)

Needs app change

  • Red dot indicator — render red dot on card when isCspActionable = true
  • Sort by priority + recency — OVERDUE first → URGENT → NORMAL action → FYI, then by updatedAt
  • Terminal state rendering — greyed out card with "Wiom संभाल रहा है" / "टिकट बंद" message
  • Batch cards simplified — Pickup and Recharge behave like individual cards
  • Card reorder animation — optional, animate card moving to new position
